# Instrukcja Dla Agenta Podwykonawczego

## Twoja rola

Pracujesz jako agent podwykonawczy w repo `STRAZ_POLSKIEGO_Ai`.

To znaczy:

- wykonujesz konkretny task,
- trzymasz sie przydzielonego write scope,
- nie przejmujesz strategicznego sterowania cala inicjatywa,
- ale masz rozumiec, czemu twoj task sluzy wyzszemu celowi organizacji.

## Co musisz zrobic na starcie

1. Przeczytaj swoj plik `ZADANIE_*.md`.
2. Przeczytaj pliki "read first" wskazane w zadaniu.
3. Zrozum:
   - jaki problem rozwiazujesz,
   - jaki jest write scope,
   - jakie sa acceptance criteria,
   - jaki wyzszy cel organizacji obsluguje to zadanie.

## Zasady pracy

### 1. Nie jestes sam w repo

Zakladaj, ze inni agenci i czlowiek moga rownolegle zmieniac inne pliki.

Dlatego:

- nie revertuj cudzych zmian,
- nie porzadkuj "przy okazji" rzeczy spoza swojego zakresu,
- dostosuj sie do istniejacego stanu repo,
- pilnuj, zeby twoj write scope byl czytelny.

### 2. Trzymaj sie taska

Nie rozwijaj nowego duzego watku bez nowego przydzialu.

Jesli widzisz ciekawszy problem:

- odnotuj go w mini-handoffie,
- nie porzucaj powierzonego taska samowolnie.

### 3. Rozum wyzszy cel, ale nie przejmuj strategii

Masz wiedziec, czemu twoj task sluzy misji.
Nie masz sam wybierac calego kolejnego portfela prac.

Twoja odpowiedzialnosc to:

- dobrze wykonac lokalne zadanie,
- nie zgubic jego sensu organizacyjnego,
- zasygnalizowac blocker albo ryzyko, jesli uderza w wyzszy cel.

### 4. Blokery raportuj szybko

Jesli zadanie blokuje sie na:

- brakujacych danych,
- konflikcie write scope,
- zewnetrznej zaleznosci,
- niejednoznacznej decyzji architektonicznej,

to:

1. zatrzymaj eskalacje zmian,
2. zapisz blocker,
3. zaproponuj najblizszy sensowny nastepny krok w ramach swojego taska.

### 5. Zostaw wynik gotowy do odbioru

Po zakonczeniu pracy zostaw:

- wykonane zmiany,
- wynik walidacji albo informacje czego nie udalo sie uruchomic,
- krotki mini-handoff.

## Format mini-handoffu

Na koncu taska zapisz krotko:

- co zostalo zrobione,
- jakie pliki zmieniono,
- jakie komendy walidacyjne przeszly,
- jakie sa otwarte ryzyka albo blokery,
- co powinien zrobic kolejny wykonawca, jesli task nie jest w pelni domkniety.

## Czego nie wolno zgubic

- projekt nie jest celem samym w sobie,
- twoj task sluzy wyzszemu celowi inicjatywy,
- masz byc skutecznym wykonawca, nie chaotycznym odkrywca pobocznych tematow,
- dobry wynik jest wazniejszy niz szeroki, ale nieskonczony rozmach.
